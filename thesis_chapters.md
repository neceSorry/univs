# РАЗРАБОТКА ВЕБ-ПРИЛОЖЕНИЯ ДЛЯ УПРАВЛЕНИЯ УЧЕБНЫМ ПРОЦЕССОМ УНИВЕРСИТЕТА

# ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

## 1.1 Описание предметной области

Управление учебным процессом в высшем учебном заведении представляет собой комплекс взаимосвязанных административных, организационных и информационных задач, которые обеспечивают функционирование образовательного учреждения. В состав этого комплекса входят: ведение контингента обучающихся и преподавательского состава, формирование учебных планов, составление расписания занятий, учёт успеваемости и посещаемости, организация экзаменационных сессий, управление финансовыми расчётами со студентами.

Традиционно значительная часть перечисленных процессов осуществляется вручную или с применением разрозненных инструментов — электронных таблиц, отдельных учётных программ, бумажной документации. Это порождает ряд системных проблем: дублирование данных, несогласованность сведений в разных отделах, высокая трудоёмкость рутинных операций, затруднённый доступ к актуальной информации для студентов и преподавателей.

В рамках данной дипломной работы разрабатывается интегрированная веб-система управления учебным процессом, которая автоматизирует перечисленные задачи в едином информационном пространстве с разграничением прав доступа по ролям.

## 1.2 Анализ существующих аналогов

### 1.2.1 1С:Университет

Программный продукт «1С:Университет» является одним из наиболее распространённых решений для автоматизации учебного процесса в русскоязычном сегменте. Система охватывает управление контингентом, успеваемостью, учебными планами и расписанием.

**Достоинства:** широкая функциональность, развитая экосистема интеграций, соответствие нормативным требованиям.

**Недостатки:** высокая стоимость лицензирования и внедрения, сложность настройки под специфику конкретного вуза, устаревший пользовательский интерфейс, зависимость от платформы 1С, отсутствие современных веб-технологий и мобильного доступа.

### 1.2.2 Moodle

Moodle — система управления обучением (LMS) с открытым исходным кодом, ориентированная преимущественно на организацию электронного обучения: публикацию материалов, тестирование, форумы.

**Достоинства:** открытый код, большое сообщество, богатая экосистема модулей.

**Недостатки:** система не предназначена для комплексного управления учебным процессом — в ней отсутствуют полноценные модули составления расписания, управления структурой вуза, расчёта оплаты, генерации расписания.

### 1.2.3 Deканат (различные реализации)

Системы класса «Деканат» автоматизируют работу деканата: ведение личных дел студентов, зачётных книжек, приказов. Как правило, это десктопные приложения с ограниченным числом пользователей.

**Недостатки:** отсутствие веб-доступа, нет личных кабинетов для студентов и преподавателей, не решают задачи составления расписания.

### 1.2.4 Сравнительный анализ

Проведённый анализ показал, что ни одно из рассмотренных решений не обеспечивает в полной мере автоматическую генерацию расписания на основе оптимизационных алгоритмов при одновременном наличии современного веб-интерфейса с ролевыми личными кабинетами. Это определяет актуальность и практическую ценность разрабатываемой системы.

## 1.3 Постановка задачи

**Цель:** разработать веб-приложение для управления учебным процессом университета, объединяющее в единой информационной системе администрирование структуры вуза, автоматическое составление расписания и личные кабинеты студентов и преподавателей.

**Задачи:**
1. Разработать модуль управления организационной структурой (институты, кафедры, направления, группы).
2. Разработать модуль управления персоналом (преподаватели, студенты).
3. Разработать модуль учебных планов с поддержкой нескольких типов занятий (лекции, практики, лабораторные).
4. Разработать сервис автоматической генерации расписания на основе генетического алгоритма с учётом предпочтений преподавателей и ограничений аудиторного фонда.
5. Разработать модуль управления версиями расписания (черновик / опубликовано).
6. Разработать личные кабинеты трёх ролей пользователей: администратора, преподавателя, студента.
7. Разработать модули учёта успеваемости, посещаемости, экзаменов и оплат.
8. Реализовать систему аутентификации и авторизации на основе JWT и RBAC.

## 1.4 Функциональные требования

### Роль «Администратор»:
- Управление иерархической структурой вуза (институты → кафедры → направления → группы).
- Управление учётными записями преподавателей и студентов.
- Формирование учебных планов по семестрам.
- Настройка потоков (объединение групп на общих лекциях).
- Запуск генерации расписания с настройкой весов ограничений.
- Просмотр и ручная корректировка сгенерированного расписания.
- Публикация расписания (переход из статуса «черновик» в «опубликовано»).
- Управление экзаменационным расписанием.
- Просмотр финансовых расчётов.
- Просмотр аналитики.

### Роль «Преподаватель»:
- Просмотр личного расписания.
- Просмотр списка своих дисциплин и прикреплённых групп.
- Заполнение журнала оценок и посещаемости.
- Управление учебными материалами.
- Заполнение предпочтений для составления расписания (недоступные дни, предпочтительные пары).
- Просмотр и ввод результатов экзаменов.

### Роль «Студент»:
- Просмотр личного профиля.
- Просмотр расписания занятий.
- Просмотр журнала оценок и посещаемости.
- Просмотр расписания экзаменов и результатов.
- Просмотр финансовой информации (задолженность, оплата).

## 1.5 Нефункциональные требования

- **Производительность:** интерфейс должен реагировать на действия пользователя не более чем через 300 мс для операций без генерации расписания.
- **Масштабируемость:** архитектура должна допускать добавление новых модулей без изменения существующих.
- **Безопасность:** все API-эндпоинты, требующие аутентификации, должны быть защищены JWT-токенами; пароли хранятся только в виде хэшей bcrypt.
- **Удобство использования:** интерфейс должен соответствовать принципам Material Design / Ant Design и быть интуитивно понятным без предварительного обучения.
- **Доступность:** система работает в браузере без установки дополнительного программного обеспечения.

# ГЛАВА 2. ПРОЕКТИРОВАНИЕ СИСТЕМЫ

## 2.1 Архитектура системы

Разрабатываемая система построена по трёхуровневой клиент-серверной архитектуре с выделением отдельного микросервиса для задачи оптимизации расписания.

```
┌─────────────────────────────────────────────────────────────────┐
│                         КЛИЕНТСКИЙ УРОВЕНЬ                       │
│           React SPA (порт 5173) — браузер пользователя           │
└──────────────────────────────┬──────────────────────────────────┘
                               │ HTTP / REST API
┌──────────────────────────────▼──────────────────────────────────┐
│                        СЕРВЕРНЫЙ УРОВЕНЬ                         │
│           NestJS API (порт 3000) — основной backend              │
│                               │                                  │
│              HTTP (внутренняя сеть, порт 8001)                   │
│                               │                                  │
│         Python FastAPI Scheduler (порт 8001) — генерация         │
└──────────────────────────────┬──────────────────────────────────┘
                               │ TCP / SQL
┌──────────────────────────────▼──────────────────────────────────┐
│                         УРОВЕНЬ ДАННЫХ                           │
│  PostgreSQL (порт 5433)  │  Redis (порт 6379)  │  MinIO (9000)  │
└─────────────────────────────────────────────────────────────────┘
```

**Взаимодействие компонентов:**
- Клиент (React SPA) обращается к NestJS API по протоколу HTTP через Axios.
- NestJS обрабатывает бизнес-логику, обращается к PostgreSQL через TypeORM и при необходимости делегирует задачу генерации расписания Python-сервису.
- Python-сервис принимает данные через REST, выполняет генетический алгоритм в фоновом потоке и возвращает готовое расписание по запросу.
- Redis зарезервирован для кэширования и управления сессиями; MinIO — для хранения учебных материалов.

Такая архитектура обеспечивает изоляцию вычислительно-ёмкой задачи (генетический алгоритм) от основного сервера, а также независимое масштабирование компонентов.

## 2.2 Выбор технологий и их обоснование

### 2.2.1 Backend — NestJS + TypeScript

NestJS — это прогрессивный фреймворк для разработки серверных приложений на Node.js, построенный поверх Express.js и вдохновлённый архитектурными концепциями Angular (модули, декораторы, внедрение зависимостей).

**Обоснование выбора:**
- Строгая модульная структура упрощает поддержку больших приложений с многочисленными доменами (в проекте 18 модулей).
- Встроенная поддержка TypeScript обеспечивает статическую типизацию, что снижает количество ошибок на этапе компиляции.
- Декораторы (`@Controller`, `@Injectable`, `@Roles`) позволяют декларативно описывать поведение без засорения бизнес-логики.
- Зрелая экосистема: встроенные Pipes (валидация DTO), Guards (авторизация), Interceptors (трансформация ответов).
- Совместимость с TypeORM, Passport, class-validator «из коробки».

### 2.2.2 ORM — TypeORM + PostgreSQL

TypeORM — ORM-библиотека для TypeScript/JavaScript, поддерживающая паттерны Active Record и Data Mapper.

**Обоснование выбора TypeORM:**
- Декораторы сущностей (`@Entity`, `@Column`, `@ManyToOne`) позволяют описывать схему базы данных прямо в классах TypeScript, что обеспечивает единственный источник истины.
- Режим `synchronize: true` автоматически приводит схему базы данных в соответствие с сущностями при запуске — удобно в процессе разработки.
- Поддержка сложных связей: OneToMany, ManyToOne, ManyToMany через промежуточные таблицы.

**Обоснование выбора PostgreSQL:**
- Полноценная реляционная СУБД с поддержкой JSON/JSONB (используется для хранения `penalty_details` расписания).
- Надёжность, ACID-транзакции, широкая поддержка в экосистеме Node.js.
- Версия 15 обеспечивает высокую производительность и расширенные возможности работы с типами данных.

### 2.2.3 Frontend — React + Vite + TypeScript

**React 19** — библиотека для построения пользовательских интерфейсов на основе компонентного подхода.

**Обоснование выбора React:**
- Декларативная модель описания UI: компоненты описывают, _что_ должно быть отрисовано, а не _как_.
- Виртуальный DOM обеспечивает эффективное обновление только изменившихся частей страницы.
- Огромная экосистема библиотек и широкое сообщество.
- Хуки (useState, useEffect, useCallback) позволяют переиспользовать логику без наследования.

**Vite** заменяет устаревший Create React App в качестве инструмента сборки. Использует нативные ES-модули браузера для мгновенного обновления при разработке (Hot Module Replacement), что значительно ускоряет цикл разработки.

### 2.2.4 Управление состоянием — Zustand + React Query

**Zustand** — минималистичная библиотека управления глобальным состоянием.

**Обоснование:** В приложении глобальным состоянием являются только данные аутентификации (токен, роль, профиль пользователя). Для этой задачи избыточно использовать Redux; Zustand обеспечивает простой API без шаблонного кода.

**React Query (TanStack Query)** управляет серверным состоянием (кэшированием, загрузкой, синхронизацией данных с API).

**Обоснование:** Вместо ручного управления флагами `isLoading/isError` и вызовами `useEffect`, React Query предоставляет декларативный API (`useQuery`, `useMutation`) с автоматическим кэшированием, инвалидацией кэша и фоновым обновлением данных.

### 2.2.5 UI-библиотека — Ant Design

Ant Design — комплексная UI-библиотека корпоративного уровня с готовыми компонентами (таблицы, формы, модальные окна, выпадающие списки, уведомления).

**Обоснование:** Для административных интерфейсов с большим количеством таблиц и форм Ant Design предоставляет готовые компоненты высокого качества, не требующие дополнительной стилизации с нуля. Это существенно ускоряет разработку.

### 2.2.6 Планировщик — Python + FastAPI + DEAP

Задача составления расписания университета относится к классу задач удовлетворения ограничений (Constraint Satisfaction Problem, CSP) и является NP-сложной. Для её решения применяется метаэвристический подход — генетический алгоритм.

**Обоснование выбора Python:**
- DEAP (Distributed Evolutionary Algorithms in Python) — специализированная библиотека для эволюционных вычислений с готовыми реализациями операторов отбора, скрещивания и мутации.
- Богатая экосистема научных библиотек.
- Отсутствие необходимости в сложных вычислениях в реальном времени — задача запускается в фоне.

**FastAPI** — современный асинхронный веб-фреймворк для Python с автоматической генерацией OpenAPI-документации и строгой типизацией через Pydantic.

### 2.2.7 Инфраструктура — Docker Compose

Docker Compose используется для оркестрации зависимых сервисов (PostgreSQL, Redis, pgAdmin, MinIO) в изолированных контейнерах. Это обеспечивает воспроизводимость среды разработки независимо от операционной системы разработчика.

## 2.3 Проектирование базы данных

### 2.3.1 Концептуальная модель данных

База данных содержит **25 сущностей**, объединённые в несколько логических групп.

**Организационная иерархия:**
```
Institute → Department → Program → Group → Student
```
Данная иерархия отражает структуру реального университета: студенты принадлежат к группам, группы — к направлениям подготовки, направления — к кафедрам, кафедры — к институтам.

**Персонал:**
- `User` — базовая учётная запись с полем `role` (admin | teacher | student).
- `Teacher` — профиль преподавателя, связанный с `User` и кафедрой `Department`.
- `Student` — профиль студента, связанный с `User` и группой `Group`.

**Учебные планы и дисциплины:**
- `Discipline` — учебная дисциплина с типом и привязкой к кафедре.
- `CurriculumPlan` — семестровый учебный план для направления.
- `CurriculumItem` — строка учебного плана: дисциплина + часы трёх типов занятий + три преподавателя.

**Расписание:**
- `ScheduleVersion` — версия расписания (черновик или опубликованная).
- `ScheduleSlot` — конкретная пара: день недели, номер пары, тип недели, аудитория, преподаватель, группа.
- `Stream` / `StreamGroup` — потоковые лекции для нескольких групп.

**Оценки и посещаемость:**
- `Grade`, `GradeBookEntry`, `Attendance`, `Exam`, `ExamResult`.

**Прочее:**
- `TeacherPreference` — предпочтения преподавателя для алгоритма.
- `Payment` — финансовые данные студента.
- `CourseMaterial` — учебные материалы.
- `AdminPermission` — детализированные права администраторов.

### 2.3.2 Ключевые проектные решения

**1. UUID в качестве первичных ключей.**
Все сущности используют `@PrimaryGeneratedColumn('uuid')`. Это обеспечивает уникальность идентификаторов независимо от узла генерации, что упрощает горизонтальное масштабирование и экспорт/импорт данных.

**2. Три преподавателя на одну строку учебного плана.**
Сущность `CurriculumItem` содержит три внешних ключа: `teacher_lecture_id`, `teacher_practice_id`, `teacher_lab_id`. Это решение отражает реальную практику, когда теоретическую часть курса ведёт один преподаватель, практические занятия — другой, а лабораторные работы — третий. Альтернативный вариант (отдельная таблица для назначений преподавателей) усложнил бы запросы без практической выгоды.

**3. Аудитории как текстовые поля, а не FK.**
Поле `preferred_classroom: string` и `classroom_text: string` хранят номер аудитории в виде строки. Это обоснованное упрощение: алгоритм проверяет конфликты по текстовому совпадению, а вместимость аудитории не учитывается (её нет в исходных данных). Сущность `Classroom` создана для хранения физических данных об аудиториях, однако в текущей версии алгоритм использует текстовые ссылки.

**4. JSONB для метаданных качества расписания.**
Поле `penalty_details` в `ScheduleVersion` хранится как PostgreSQL JSONB. Это позволяет сохранять произвольную структуру нарушений (количество конфликтов преподавателей, окон, перегрузок) без создания отдельных таблиц.

**5. Поддержка чётных/нечётных недель.**
`ScheduleSlot.week_type` принимает значения `all | odd | even`. Это позволяет составлять расписание с чередованием: в нечётные недели одна пара, в чётные — другая. Особенно важно для семестров с большим объёмом часов.

### 2.3.3 Описание основных сущностей

**Таблица 2.1 — Сущность User**

## 2.6 Проектирование алгоритма генерации расписания

### 2.6.1 Постановка задачи оптимизации

Задача составления расписания университета формально является задачей удовлетворения ограничений (CSP). Необходимо распределить N учебных занятий по слотам (день × пара × тип недели) таким образом, чтобы:
- ни один преподаватель не вёл два занятия одновременно (жёсткое ограничение);
- ни одна группа не посещала два занятия одновременно (жёсткое ограничение);
- ни одна аудитория не использовалась дважды в один момент (жёсткое ограничение);
- преподаватель не назначался в недоступные дни (жёсткое ограничение);
- нарушения мягких ограничений (окна в расписании, перегрузка) были минимальны.

Задача является NP-сложной, поэтому точный алгоритм нецелесообразен при реальных размерах данных. Применяется **генетический алгоритм** — метаэвристика, моделирующая естественный отбор.

### 2.6.2 Структура хромосомы

Каждый индивид популяции представляет собой список генов. Каждый **ген** соответствует одному занятию и содержит:
- идентификаторы дисциплины, группы, преподавателя;
- тип занятия (лекция, практика, лабораторная);
- аудиторию (текстовая строка);
- временной слот: день недели (1–6), номер пары (1–6), тип недели (all/odd/even);
- признак потоковой лекции и список групп потока.

Число генов определяется суммарным количеством занятий всех групп на семестр. Например, дисциплина с 36 часами лекций порождает 1 ген на лекцию (36/36 ≈ 1 занятие в неделю).

### 2.6.3 Функция приспособленности

Функция приспособленности оценивает расписание по сумме штрафов:

**Жёсткие ограничения** (вес штрафа = 1000):
1. Конфликт преподавателя: два занятия одного преподавателя в один временной слот.
2. Конфликт группы: две пары одной группы в один слот.
3. Конфликт аудитории: два занятия в одной аудитории в один слот.
4. Недоступный день: преподаватель назначен в день из списка `unavailable_days`.

**Мягкие ограничения:**
5. Нежелательное время (вес 10): занятие не в `preferred_periods`.
6. Суточная перегрузка преподавателя (вес 30): превышение `max_periods_per_day`.
7. Окна в расписании преподавателя (вес 20): несмежные пары в одном дне.
8. Превышение нагрузки за неделю (вес 50): превышение `max_periods_per_week`.
9. Окна в расписании группы (вес 15).
10. Суточная перегрузка группы (вес 25): более 4 пар в день.

Качество расписания: `score = 100 × (1 − total_penalty / max_penalty)`.

### 2.6.4 Генетические операторы

- **Инициализация:** случайное назначение временных слотов для каждого гена.
- **Отбор:** турнирный отбор (размер турнира = 3) — выбирается лучший из трёх случайных индивидов.
- **Скрещивание:** двухточечное скрещивание на списке генов с вероятностью 0.7.
- **Мутация:** случайное изменение дня или номера пары у случайного гена с вероятностью 0.05.

**Параметры алгоритма:** размер популяции 150, число поколений 300.

### 2.6.5 Асинхронное выполнение

Генетический алгоритм выполняется в фоновом потоке FastAPI (`BackgroundTasks`). NestJS запускает задачу через `POST /generate`, получает `task_id` и опрашивает `/status/{task_id}` с интервалом 2 секунды вплоть до завершения (тайм-аут 5 минут).

### 3.2.2 Реализация аутентификации (JWT)

Аутентификация реализована с использованием стандарта JWT (JSON Web Token). При входе в систему сервер выпускает два токена:
- **Access token** — срок действия 15 минут; передаётся в заголовке каждого запроса.
- **Refresh token** — срок действия 7 дней; хранится в `localStorage` клиента и используется исключительно для получения новой пары токенов.

**Процесс входа** (`AuthService.login`):
1. Клиент отправляет `POST /auth/login` с `{ email, password }`.
2. Сервис находит пользователя в базе по email.
3. Сравнивает пароль с хэшем через `bcrypt.compare()`.
4. При совпадении формирует JWT-payload: `{ sub, email, role, is_super_admin, permissions }`.
5. Возвращает `{ access_token, refresh_token, user }`.

**JWT-стратегия** (`jwt.strategy.ts`) реализует интерфейс `PassportStrategy`. При каждом запросе к защищённому эндпоинту Passport извлекает токен из заголовка `Authorization: Bearer <token>`, верифицирует подпись и передаёт декодированный payload в `request.user`.

### 3.2.3 Реализация авторизации (RBAC)

Авторизация реализована через Guards — перехватчики NestJS, выполняемые до контроллера.

**Декоратор `@Roles`** задаёт требуемые роли для эндпоинта:

```typescript
// university-backend/src/common/decorators/roles.decorator.ts
export const Roles = (...roles: string[]) => SetMetadata('roles', roles);
```

**`RolesGuard`** читает метаданные декоратора и проверяет соответствие:

```typescript
canActivate(context: ExecutionContext): boolean {
  const requiredRoles = this.reflector.get<string[]>('roles', context.getHandler());
  const user = context.switchToHttp().getRequest().user;

  // Суперадмин обходит все проверки ролей
  if (user.is_super_admin) return true;

  // Проверка соответствия роли
  if (requiredRoles && !requiredRoles.includes(user.role)) return false;

  // Проверка детальных прав (если указаны)
  const requiredPermission = this.reflector.get<string>('permission', context.getHandler());
  if (requiredPermission) {
    return user.permissions?.includes(requiredPermission) ?? false;
  }

  return true;
}
```

Защита эндпоинта в контроллере:

```typescript
@Post()
@Roles('admin')
@UseGuards(JwtAuthGuard, RolesGuard)
createStudent(@Body() dto: CreateStudentDto) { ... }
```

- Возвращает метрику готовности к генерации.

**Генерация расписания** (`POST /schedule/generate`):
1. Повторяет сбор данных, но теперь формирует полный payload для Python-сервиса.
2. **Объединяет лекции в потоки:** если несколько групп имеют одинакового лектора по одной дисциплине, создаётся один элемент-поток вместо N отдельных.

5. При получении статуса `done` запрашивает результат `/result/{task_id}`.
6. Создаёт `ScheduleVersion` со статусом `draft` и сохраняет `quality_score`.
7. Для каждого слота из результата создаёт записи `ScheduleSlot`. Потоковые слоты раскрываются: один слот потока превращается в N слотов — по одному на каждую группу потока.

**Ручная правка слота** (`PUT /schedule/:slotId`):
- Обновляет поля `day_of_week`, `period_number`, `classroom_text` и др.
- Устанавливает `is_manual_override = true` и сохраняет `override_reason`.

### 3.3.2 Реализация генетического алгоритма

Алгоритм в `genetic.py` использует библиотеку DEAP и реализует следующую последовательность:

**Инициализация DEAP:**
```python
creator.create("FitnessMin", base.Fitness, weights=(-1.0,))
creator.create("Individual", list, fitness=creator.FitnessMin)

toolbox = base.Toolbox()
toolbox.register("individual", init_individual, items=expanded_items)
toolbox.register("population", tools.initRepeat, list, toolbox.individual)
toolbox.register("evaluate", evaluate, weights=weights)
toolbox.register("mate", tools.cxTwoPoint)
toolbox.register("mutate", mutate_individual)
toolbox.register("select", tools.selTournament, tournsize=3)
```

**Основной цикл:**
```python
pop = toolbox.population(n=150)
for gen in range(300):
    offspring = algorithms.varAnd(pop, toolbox, cxpb=0.7, mutpb=0.05)
    fits = toolbox.map(toolbox.evaluate, offspring)
    for fit, ind in zip(fits, offspring):
        ind.fitness.values = fit
    pop = toolbox.select(offspring, k=len(pop))
best = tools.selBest(pop, k=1)[0]
```

**Функция мутации** случайно изменяет день или пару у одного гена:
```python
def mutate_individual(individual):
    idx = random.randrange(len(individual))
    gene = individual[idx].copy()
    if random.random() < 0.5:
        gene['day_of_week'] = random.randint(1, 6)
    else:
        gene['period'] = random.randint(1, 6)
    individual[idx] = gene
    return (individual,)
```

**Подсчёт нарушений** проходит по всем парам генов и проверяет совпадение слотов:
```python
def evaluate(individual, weights):
    penalty = 0
    slots_by_time = defaultdict(list)
    for gene in individual:
        key = (gene['day_of_week'], gene['period'], gene['week_type'])
        slots_by_time[key].append(gene)

    for key, genes in slots_by_time.items():
        teachers = [g['teacher_id'] for g in genes]
        if len(teachers) != len(set(teachers)):  # дублирующиеся преподаватели
            penalty += weights['hard_conflict']
        # ... аналогично для групп и аудиторий
    return (penalty,)
```

### 3.3.3 Формат результата

По завершении алгоритм возвращает:
```python
{
  "quality_score": 87.3,
  "penalty_details": {
    "hard_conflicts": 0,
    "teacher_preferred_time_violations": 4,
    "teacher_overloads": 1,
    ...
  },
  "slots": [ ... ]
}
```

`quality_score = 100 × (1 − total_penalty / max_possible_penalty)` — нормированный показатель качества от 0 до 100.

### 3.4.2 Управление состоянием аутентификации (Zustand)

```typescript
// university-frontend/src/store/auth.store.ts
interface AuthState {
  user: User | null;
  token: string | null;
  isAuthenticated: boolean;
  login: (data: LoginResponse) => void;
  logout: () => void;
}

const useAuthStore = create<AuthState>((set) => ({
  user: null,
  token: localStorage.getItem('access_token'),
  isAuthenticated: !!localStorage.getItem('access_token'),
  login: (data) => {
    localStorage.setItem('access_token', data.access_token);
    localStorage.setItem('refresh_token', data.refresh_token);
    set({ user: data.user, token: data.access_token, isAuthenticated: true });
  },
  logout: () => {
    localStorage.clear();
    set({ user: null, token: null, isAuthenticated: false });
  },
}));
```

### 3.4.3 HTTP-клиент с автоматическим обновлением токена

Центральный Axios-клиент (`api/api.client.ts`) сконфигурирован с двумя перехватчиками:

**Request Interceptor** добавляет токен к каждому запросу:
```typescript
apiClient.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
```

**Response Interceptor** обрабатывает истечение токена:
```typescript
apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retry) {
      error.config._retry = true;
      const refreshToken = localStorage.getItem('refresh_token');
      const { data } = await apiClient.post('/auth/refresh', { refresh_token: refreshToken });
      useAuthStore.getState().login(data);
      error.config.headers.Authorization = `Bearer ${data.access_token}`;
      return apiClient(error.config);  // повтор исходного запроса
    }
    return Promise.reject(error);
  }
);
```

Флаг `_retry` предотвращает бесконечный цикл: если повторный запрос тоже вернёт 401, перехватчик перенаправит на страницу входа.

### 3.4.4 Паттерн работы с данными (React Query)

Все запросы к API выполняются через React Query. Пример страницы списка студентов:

```typescript
// Получение данных
const { data, isLoading } = useQuery({
  queryKey: ['students', { groupId }],
  queryFn: () => apiClient.get('/students', { params: { groupId } }).then(r => r.data.data),
});

// Создание записи
const createMutation = useMutation({
  mutationFn: (dto: CreateStudentDto) => apiClient.post('/students', dto),
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['students'] }); // инвалидация кэша
    message.success('Студент создан');
    setModalOpen(false);
  },
  onError: (err) => message.error('Ошибка создания'),
});
```

`invalidateQueries` после мутации автоматически перезапрашивает список, обеспечивая актуальность данных без ручного управления состоянием.
